# ADR-001：采用前后端分离结构

## 背景

CodebaseCoach 是一个开源项目学习与面试准备 Agent。前端需要提供仓库地址输入、页面路由、文件树、Agent 步骤、工具日志和 Markdown 预览；后端后续需要处理 GitHub URL 解析、仓库 clone、文件读取、AI 调用、Markdown 文档保存和历史记录。

第一阶段只实现项目骨架，但前后端边界会影响后续所有阶段。如果边界不清晰，API Key、本地文件系统访问和 AI 调用可能被错误放到浏览器端。

## 备选方案

### 方案 A：前后端分离

前端放在 `apps/web`，后端放在 `apps/server`，两者通过 HTTP API 通信。

优点是职责边界清晰，API Key 和本地文件系统能力留在后端，前端和后端可以独立启动、构建和演进。缺点是本地开发需要同时启动两个服务，并维护 CORS 与 API 地址配置。

### 方案 B：单体应用

把页面和服务端逻辑放在一个应用里。

优点是入口较少，原型初期看起来更简单。缺点是后续仓库读取、AI 调用、文档落盘和页面逻辑容易混在一起，不符合当前 Vue 3 + Vite 前端与 FastAPI 后端的技术边界。

### 方案 C：纯前端应用

只实现 Vue 前端，在浏览器中直接调用外部 API。

优点是工程最轻。缺点是无法安全保存 OpenAI API Key，也不适合执行 Git clone、扫描本地文件和保存 Markdown 文档，与“前端不直接请求 OpenAI API”的约束冲突。

## 决策

采用前后端分离结构：

- `apps/web`：Vue 3 + TypeScript + Vite 前端。
- `apps/server`：Python + FastAPI 后端。
- `generated_docs/`、`temp_repos/`、`data/`：后续由服务端使用的运行数据目录。

## 原因

前后端分离可以把用户交互和服务端能力隔离开。前端只负责展示和交互，后端统一处理环境变量、仓库文件、AI 调用和文档保存。

这种结构也更适合阶段式开发。第一阶段验证两端能独立启动和构建；后续阶段再逐步补充 repo、agent、docs、history 等 API。

## 代价

- 本地开发需要分别启动前端和后端。
- 需要维护 API 合约和跨域配置。
- 后续前端需要统一管理后端服务地址，避免硬编码散落在页面中。

## 结果

当前项目已经创建 `apps/web` 和 `apps/server`。前端页面骨架和后端 `/health` 接口已完成，真实 GitHub clone、AI 调用和文档生成未在第一阶段实现。
