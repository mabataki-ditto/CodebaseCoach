# ADR-005：为什么使用 GitPython 进行仓库读取

## 背景

第二阶段需要实现公开 GitHub 仓库 clone、本地目录树生成和基础文件读取。后续阶段还会继续做核心文件筛选、上下文构建和 AI 文档生成，这些能力都依赖对仓库工作区文件的稳定访问。

## 备选方案

### 方案 A：使用 GitPython 浅克隆仓库

通过 GitPython 调用本机 Git，将公开仓库 clone 到 `temp_repos/`，再基于本地文件系统扫描目录和读取基础文件。

优点是后续可以用统一的本地文件路径处理目录树、基础文件、核心文件筛选和文档生成。缺点是需要本机安装 Git，clone 受网络环境影响，也需要管理临时目录。

### 方案 B：使用 GitHub REST API

通过 GitHub API 获取仓库内容和文件树。

优点是不需要本地 Git，也可以按 API 分页获取内容。缺点是要处理 API rate limit、目录递归、文件内容编码和认证问题；后续如果要分析大量文件，仍然会变成多次 API 请求。

### 方案 C：只解析 URL，不读取仓库

只返回 owner/repo，不做 clone 和文件读取。

优点是实现简单。缺点是无法满足第二阶段“返回目录树”和“基础文件摘要”的验收，也无法支撑后续核心文件筛选。

## 决策

第二阶段使用 GitPython 对公开仓库进行浅克隆：

- clone 目标目录：`temp_repos/{owner}_{repo}_{timestamp}/`。
- clone 方式：`depth=1`。
- clone 后基于本地文件系统生成目录树和读取基础文件。

## 原因

项目后续的核心流程是“读取仓库 -> 筛选文件 -> 构建上下文 -> 生成文档”。本地工作区更适合做文件过滤、大小限制和核心文件筛选，也更贴近真实开发者分析仓库的方式。

使用浅克隆可以降低网络和磁盘成本。当前阶段不需要提交历史，因此没有必要完整克隆仓库。

## 代价

- 运行环境必须有 Git。
- clone 公开仓库需要网络访问，可能因为代理、证书或凭据环境失败。
- 需要后续补充临时目录清理策略。

## 结果

当前实现已通过 GitPython clone 公开仓库到 `temp_repos/`。当 clone 失败时，后端返回结构化 `REPO_CLONE_FAILED`，不会导致服务崩溃。
