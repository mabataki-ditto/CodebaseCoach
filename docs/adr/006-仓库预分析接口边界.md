# ADR-006：仓库预分析接口边界

## 背景

第二阶段需要给前端提供仓库预分析接口。预分析需要返回 owner、repo、目录树和基础文件摘要，但不能真实调用 AI，不能生成最终文档，也不能一次性读取整个仓库内容。

## 备选方案

### 方案 A：拆分 `/api/repo/parse` 和 `/api/repo/scan`

`parse` 只做 URL 解析；`scan` 执行解析、clone、目录树生成和基础文件摘要读取。

优点是轻量校验和重型扫描边界清楚。缺点是前端需要理解两个接口的不同用途。

### 方案 B：只提供一个 `/api/repo/scan`

所有操作都放进一个接口。

优点是接口数量少。缺点是前端无法低成本只做 URL 解析；clone 失败会和 URL 解析失败混在一个调用里。

### 方案 C：扫描接口返回完整文件内容

clone 后读取并返回大量文件内容。

优点是后续 AI 分析可能少一次读取。缺点是违反“不要一次性读取所有文件内容”和“必须限制单文件读取大小”的约束，也会让响应体过大。

## 决策

采用两个接口：

- `POST /api/repo/parse`：只解析 URL，返回 `owner`、`repo`、`repo_url`。
- `POST /api/repo/scan`：clone 公开仓库，返回 `owner`、`repo`、`repo_url`、`file_tree`、`basic_files`。

扫描接口只返回基础文件摘要，不返回本地绝对路径，不返回所有文件内容。

## 原因

这个边界符合前端预分析需求，也能控制后端工作量。URL 解析是纯计算，可以快速失败；仓库扫描涉及网络和磁盘，需要单独处理错误和耗时。

不返回本地绝对路径可以避免泄露运行环境信息。只返回基础文件摘要可以减少响应体，也为后续核心文件筛选阶段保留空间。

## 代价

- 前端需要先决定调用 `parse` 还是 `scan`。
- 后续如果要展示扫描进度，可能需要新增任务化接口或 SSE。
- 当前目录树有深度和节点数限制，不能代表完整仓库内容。

## 结果

当前 `parse`、`scan` 接口已实现。无效 URL 返回 400，clone 失败返回结构化 502。用户示例仓库 `modelcontextprotocol/typescript-sdk` 的扫描可以返回目录树和 `package.json`、`README.md` 摘要。
