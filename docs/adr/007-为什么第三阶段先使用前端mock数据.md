# ADR-007：为什么第三阶段先使用前端 mock 数据

## 背景

第三阶段目标是分析工作台 UI 和 mock 数据。第二阶段后端已经具备 URL 解析和仓库预扫描能力，但完整 Agent 步骤、工具日志、核心文件筛选、AI 文档生成和历史记录仍未完成。

如果第三阶段直接绑定真实后端全流程，会把 UI 工作卡在后续能力上，也容易在前端写死临时业务逻辑。

## 备选方案

### 方案 A：前端使用 mock 数据搭建工作台

在前端集中维护 mock 文件树、核心文件、Agent 步骤、工具日志和 Markdown 预览数据，用于完成工作台布局和演示。

优点是第三阶段可以独立验证 UI 信息架构；后续只需替换数据来源。缺点是 mock 数据需要明确标注，避免被误认为真实分析结果。

### 方案 B：直接绑定第二阶段 `/api/repo/scan`

工作台直接调用真实扫描接口。

优点是可以展示真实目录树和基础文件。缺点是当前后端还没有核心文件筛选、Agent 步骤、工具日志和文档生成，页面仍要补大量前端临时逻辑。

### 方案 C：等待完整 Agent/AI 流程后再做 UI

等后端完整后再开发工作台。

优点是数据更真实。缺点是 UI 信息架构反馈太晚，后续可能发现页面布局和数据结构不匹配，返工成本更高。

## 决策

第三阶段先使用前端 mock 数据完成工作台 UI：

- mock 数据集中放在 `apps/web/src/mocks/workspace.ts`。
- 工作台页面放在 `apps/web/src/pages/WorkspacePage.vue`。
- 页面明确标注“Mock 数据”和“未调用 AI”。
- 不调用真实 AI，不生成最终文档。

## 原因

本阶段的核心风险是 UI 信息架构，而不是后端能力。mock 数据可以让前端先验证文件树、核心文件、Agent 步骤、工具日志和 Markdown 预览是否能组成一个清晰工作台。

把 mock 数据集中管理，也能降低后续替换为 API 响应的成本。

## 代价

- 当前页面展示的是模拟流程，不能代表真实分析结果。
- 后续接入真实 API 时，需要把 mock 数据替换为接口响应和状态管理。
- 仍需要补浏览器截图、控制台和移动视口验证。

## 结果

当前工作台能展示 mock 文件树、核心文件、Agent 步骤、工具日志和 Markdown 文档预览。`pnpm build` 通过，临时 dev server 访问 `/workspace` 返回 200。
