# ADR-008：核心文件筛选采用规则优先策略

## 背景

第四阶段需要在不调用真实 AI 的前提下，从 clone 后的仓库中筛选 5 到 12 个核心文件，并读取有限长度的内容摘要。项目同时要求不要一次性读取所有文件，不要读取二进制文件，不要引入 LangChain、MCP 或 RAG。

如果这个阶段就依赖模型判断核心文件，会让本地演示依赖 API Key，也会把后续 AI 接入阶段提前。

## 备选方案

### 方案 A：规则优先筛选

使用文件名、入口文件命名和目录关键词打分，优先选择 README、package.json、pyproject.toml、requirements.txt、main、index、app、server，以及 `src`、`app`、`core`、`agent`、`tools`、`services`、`api`、`routes` 目录下的文件。

优点是实现简单、结果可解释、单元测试容易覆盖，不需要 API Key。缺点是无法理解文件语义，可能漏掉命名不典型但实际重要的模块。

### 方案 B：AI 判断核心文件

把目录树交给模型，由模型选择核心文件。

优点是语义判断能力更强。缺点是违反当前阶段“不调用真实 AI”的约束，也会让无 API Key 演示流程不可控。

### 方案 C：读取全部源码后再筛选

先读取仓库所有文本文件，再根据内容筛选。

优点是信息更完整。缺点是违反“不一次性读取所有文件内容”和“必须限制单文件读取大小”的约束，性能和响应体都会失控。

## 决策

第四阶段采用规则优先筛选：

- 在 `apps/server/app/services/file_selector_service.py` 中实现。
- 根目录基础文件优先级最高。
- 入口文件和核心目录文件次之。
- 过滤依赖、构建产物、缓存、测试目录和 Git 元数据。
- 只读取文本文件，单文件按 `MAX_CORE_FILE_BYTES` 截断。
- 最多返回 `MAX_CORE_FILES` 个核心文件，当前默认 12 个。

## 原因

规则优先策略最符合当前 MVP 阶段目标：可运行、可解释、无 API Key 可演示。它也能为后续真实 AI 接入提供受控上下文，避免把整个仓库直接塞给模型。

规则本身可以用单元测试验证，便于在不同仓库上持续迭代。

## 代价

- 规则只能基于路径、文件名和有限摘要判断，不能完全理解业务语义。
- 多语言、多包仓库中可能出现嵌套 README 或配置文件竞争，需要继续调整权重。
- 后续接入真实 AI 后，可能需要把规则筛选与模型辅助筛选结合。

## 结果

当前接口 `POST /api/agent/analyze/mock` 可以返回后端筛选出的核心文件。单元测试覆盖了优先级、目录过滤、二进制跳过和内容截断。真实仓库验收中成功筛选出 12 个核心文件。
