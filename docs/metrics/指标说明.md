# CodebaseCoach 工程指标说明

## 1. 目标

本目录用于记录 CodebaseCoach 在仓库解析和扫描过程中的轻量工程指标，为后续 README、benchmark report 和简历量化表达提供真实数据来源。

当前指标只覆盖：

- `POST /api/repo/parse`
- `POST /api/repo/scan`
- `POST /api/agent/analyze/mock`

当前实现不是完整监控系统，不提供指标查询 API，不引入数据库，也不接入 Prometheus / OpenTelemetry。

## 2. 存储位置

默认写入：

```text
data/metrics.jsonl
```

每次仓库解析或扫描追加一行 JSON。JSONL 便于后续用脚本、表格或 benchmark report 直接统计。

配置项：

```env
METRICS_FILE=../../data/metrics.jsonl
```

## 3. 数据结构

后端 schema：

```text
apps/server/app/schemas/metrics.py
```

服务封装：

```text
apps/server/app/services/metrics_service.py
```

字段说明：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `record_id` | string | 单条指标记录 ID，使用 UUID hex。 |
| `operation` | `repo_parse` / `repo_scan` | 本次记录对应的操作类型。 |
| `status` | `success` / `failed` | 操作是否成功。 |
| `repo_url` | string | 规范化后的 GitHub 仓库地址；失败且无法解析时保留原始输入。 |
| `owner` | string / null | GitHub owner，解析失败时为空。 |
| `repo` | string / null | GitHub repo，解析失败时为空。 |
| `started_at` | datetime | 操作开始时间。 |
| `ended_at` | datetime | 操作结束时间。 |
| `duration_ms` | number | 操作耗时，毫秒。 |
| `cloned` | boolean | scan 是否已经完成 clone。 |
| `file_tree_node_count` | number | scan 返回的文件树节点数量，包含嵌套节点。 |
| `basic_file_count` | number | scan 读取到的基础文件数量。 |
| `basic_file_bytes` | number | scan 读取到的基础文件原始大小总和。 |
| `candidate_core_files` | number | 进入核心文件候选池的文件数量，二进制文件不计入。 |
| `selected_core_files` | number | 最终被选为核心文件的数量。 |
| `read_files` | number | 实际读取内容并进入上下文的核心文件数量。 |
| `truncated_files` | number | 因超过 `MAX_CORE_FILE_BYTES` 被截断的核心文件数量。 |
| `raw_candidate_chars` | number | 候选核心文件如果全部进入上下文的真实字符数总和。 |
| `final_context_chars` | number | 最终进入 mock / AI 上下文的字符数。 |
| `context_compression_ratio` | number | 上下文压缩比例，计算方式为 `final_context_chars / raw_candidate_chars`；当 `raw_candidate_chars = 0` 时返回 0。 |
| `mock_doc_count` | number | mock 模式实际生成并保存的 Markdown 文档数量。 |
| `mock_doc_total_chars` | number | mock 模式实际生成并保存的 Markdown 文档总字符数。 |
| `analysis_duration_ms` | number | `/api/agent/analyze/mock` 完整分析流程耗时，毫秒。 |
| `error_code` | string / null | 失败时的结构化错误码。 |
| `error_message` | string / null | 失败时的简短错误信息，不保存完整 traceback。 |

## 4. mock 分析指标计算方式

`POST /api/agent/analyze/mock` 响应中会返回完整 `metrics` 字段，同时会向 `data/metrics.jsonl` 追加一条 `operation = agent_analyze_mock` 的记录。

计算方式：

| 指标 | 计算方式 |
| --- | --- |
| `candidate_core_files` | 核心文件规则筛选后、可读取且非二进制的候选文件数量。 |
| `selected_core_files` | 最终返回到 `core_files` 的文件数量，受 `MAX_CORE_FILES` 限制。 |
| `read_files` | 当前等于 `selected_core_files`，表示已读取内容摘要的核心文件数。 |
| `truncated_files` | `core_files` 中 `truncated = true` 的数量。 |
| `raw_candidate_chars` | 所有候选核心文件完整文本解码后的字符数总和。 |
| `final_context_chars` | 最终选中文件的 `content_preview` 字符数总和。 |
| `context_compression_ratio` | `final_context_chars / raw_candidate_chars`，分母为 0 时返回 0。 |
| `mock_doc_count` | `save_markdown_documents()` 返回的文档数量。 |
| `mock_doc_total_chars` | 已保存 Markdown 文档内容字符数总和。 |
| `analysis_duration_ms` | mock 分析 workflow 从开始到 metrics 汇总前的总耗时。 |

示例响应片段：

```json
{
  "metrics": {
    "candidate_core_files": 42,
    "selected_core_files": 12,
    "read_files": 12,
    "truncated_files": 3,
    "raw_candidate_chars": 184000,
    "final_context_chars": 36000,
    "context_compression_ratio": 0.1956,
    "mock_doc_count": 7,
    "mock_doc_total_chars": 6100,
    "analysis_duration_ms": 4200
  }
}
```

## 5. 可量化表达示例

后续 README 或 benchmark report 可以基于 `data/metrics.jsonl` 统计：

- 平均仓库 URL 解析耗时。
- 仓库扫描平均耗时。
- 单次扫描平均文件树节点数。
- 单次扫描平均基础文件数量。
- clone 失败率。
- 无效 GitHub URL 失败率。
- mock 分析平均候选核心文件数量。
- 平均上下文压缩比例。
- 平均 mock 文档生成数量和文档字符数。
- mock 分析完整耗时。

简历表达示例：

```text
为仓库解析与扫描链路增加 JSONL 工程指标记录，覆盖耗时、文件树节点数、基础文件数量和失败错误码，为后续 benchmark 和项目复盘提供可量化数据来源。
```

进一步量化表达示例：

```text
为 mock 分析链路补充核心文件候选数、上下文压缩比例、文档数量和分析耗时等指标，使项目具备可复盘的 benchmark 数据来源。
```

## 6. 当前边界

- 不记录 API Key、Token、环境变量或完整异常堆栈。
- 不记录完整文件内容。
- 不统计真实 AI 调用指标；真实模型指标应在后续 `llm_service.py` 接入阶段单独设计。
- 不提供前端展示页面；当前只作为后端本地数据源。
- 不保证跨进程高并发写入的强一致性；MVP 阶段按本地单进程开发服务使用。
- 不统计 token，`context_compression_ratio` 只基于真实字符数。

## 7. 后续扩展方向

- 增加 benchmark report 脚本，从 JSONL 统计 p50 / p95 / max。
- 在历史记录阶段把 `metrics.record_id` 与分析记录关联。
- 在真实 AI 阶段增加 token、模型、mock/real 模式和 LLM 调用耗时指标。
