# CodebaseCoach 产品设计文档

说明：

开发时应以本文档作为功能边界、数据边界和交互规则依据。若界面设计、阶段提示词与本文档冲突，功能逻辑以本文档为准；视觉表现以对应页面的 UI 设计和实际实现为准。

---

## 一、项目定位

项目名称：CodebaseCoach

中文名：开源项目学习与面试准备 Agent

项目类型：面向前端开发者、AI 应用开发实习求职者的 GitHub 开源项目分析与学习文档生成工具。

核心目标：

让用户输入一个 GitHub 仓库地址后，系统自动分析项目结构、核心文件、技术栈、关键流程，并通过 AI 生成可学习、可复盘、可写进简历、可用于面试讲述的 Markdown 文档。

系统不应做成普通聊天机器人，也不应做成完整 IDE。核心是：

* GitHub 仓库解析
* 目录结构展示
* 核心文件识别
* Agent 执行过程可视化
* AI 生成学习文档
* AI 生成面试问答
* AI 生成简历描述
* Markdown 文档存储、预览与导出
* 分析历史记录
* 模型调用状态与错误提示

项目的差异化重点不是“AI 能不能分析代码”，而是把 AI 分析能力产品化为一个稳定、结构化、可解释、可复用的求职学习工作流。

---

## 二、核心用户

### 1. 主要用户

正在准备中厂 AI 应用开发、前端开发、全栈开发实习岗位的学生或初级开发者。

用户特点：

* 有一定 Vue / 前端基础
* 对 Agent、LLM、MCP、RAG 等概念了解不深
* 想通过开源项目学习工程经验
* 想将项目理解转化为简历描述和面试回答
* 希望 AI 不只是回答问题，而是帮自己沉淀学习资料

### 2. 使用场景

用户可能输入：

* 一个想学习的 GitHub 开源项目
* 一个准备贡献 PR 的仓库
* 一个准备写进简历的参考项目
* 一个面试前需要快速理解的项目

系统输出：

* 项目整体说明
* 技术栈分析
* 核心模块分析
* 核心执行流程
* 值得学习的设计点
* 面试高频问题
* 回答思路
* 简历项目描述
* 可贡献 PR 方向

## 三、核心页面与模块

当前主要模块：

* 首页
* 分析工作台
* 生成文档页
* 历史记录页
* 设置页

首页负责项目介绍、仓库输入和最近分析入口。

分析工作台负责展示仓库目录、核心文件、Agent 执行步骤、工具调用日志和分析进度。

生成文档页负责 Markdown 文档预览、切换、复制、下载和后续编辑。

历史记录页负责展示用户历史分析过的仓库和生成文档。

设置页负责模型配置、后端连接配置、显示偏好和项目说明。

## 四、首页功能逻辑

首页用于承载项目入口和核心转化路径。

首页必须包含：

* 项目名称和一句话定位
* GitHub 仓库地址输入框
* 开始分析按钮
* 示例仓库快捷入口
* 最近分析记录入口
* 项目能力说明卡片

用户输入 GitHub 仓库地址后，点击开始分析，系统进入分析工作台。

输入校验规则：

* 必须是 GitHub 仓库 URL
* 支持 `https://github.com/owner/repo`
* 支持 `https://github.com/owner/repo.git`
* 暂不要求支持私有仓库
* 暂不要求支持指定分支

校验失败时应提示用户修改，不要向后端发起无效分析请求。

首页不应做成普通聊天框。

首页的主要目标是引导用户进入“仓库分析工作流”。

## 五、分析工作台功能逻辑

分析工作台是项目核心页面。

页面结构建议：

* 顶部：仓库信息、重新分析、导出按钮
* 左侧：仓库文件树和核心文件列表
* 中间：Agent 执行步骤和工具调用日志
* 右侧：当前生成结果预览
* 底部或侧边：面试问题和简历描述入口

### 1. 仓库信息区

展示内容：

* 仓库名称
* owner / repo
* GitHub URL
* 默认分支，若无法获取则显示“不确定”
* 分析状态
* 分析开始时间和完成时间

### 2. 文件树区域

文件树来源于后端扫描结果。

必须过滤：

* `.git`
* `node_modules`
* `dist`
* `build`
* `coverage`
* `.venv`
* `venv`
* `__pycache__`
* `.next`
* `.nuxt`
* `.cache`
* `.idea`
* `.vscode`

文件树不应一次性展示过深内容。

建议默认展开：

* 根目录
* `src/`
* `app/`
* `packages/`
* `apps/`
* `server/`
* `client/`
* `core/`
* `agent/`
* `tools/`

### 3. 核心文件列表

核心文件由后端根据规则筛选。

优先级较高的文件包括：

* `README.md`
* `package.json`
* `pyproject.toml`
* `requirements.txt`
* `tsconfig.json`
* `vite.config.ts`
* `src/main.ts`
* `src/index.ts`
* `src/App.vue`
* `app/main.py`
* `app.py`
* `server.py`
* 包含 `agent`、`tools`、`services`、`api`、`routes`、`core` 的路径

每个核心文件需要展示：

* 文件路径
* 文件类型
* 选择原因
* 是否已读取
* 是否用于 AI 上下文

不要把所有文件都交给 AI。必须先筛选，再读取，再分析。

## 六、Agent 执行过程规则

本项目第一版采用“可控工作流 Agent”，不是完全自主 Agent。

核心流程：

```text
1. 解析 GitHub URL
2. 克隆或读取仓库
3. 生成目录结构
4. 读取基础文件
5. 筛选核心文件
6. 读取核心文件内容
7. 构建 AI 分析上下文
8. 生成项目概览
9. 生成核心模块解析
10. 生成面试问答
11. 生成简历描述
12. 保存 Markdown 文档
13. 返回前端展示
```

前端需要展示 Agent 执行步骤。

步骤状态包括：

* `pending`
* `running`
* `success`
* `failed`
* `skipped`

每个步骤应至少包含：

* 步骤标题
* 当前状态
* 简短说明
* 开始时间
* 结束时间
* 错误信息，若存在

工具调用日志包括：

* 工具名称
* 输入摘要
* 输出摘要
* 执行状态
* 耗时
* 错误信息

Agent 执行过程必须可解释，用户应能知道系统读了哪些文件、为什么读这些文件、生成内容基于哪些上下文。

## 七、生成文档规则

系统需要生成 Markdown 文档。

第一版至少生成：

* `01-项目概览.md`
* `02-技术栈分析.md`
* `03-核心模块解析.md`
* `04-核心流程说明.md`
* `05-面试问题与回答.md`
* `06-简历描述.md`
* `07-可贡献PR方向.md`

生成内容必须满足：

* 使用 Markdown
* 引用具体文件路径
* 不编造未提供的文件
* 信息不足时明确写“不确定”
* 区分事实、推测和建议
* 面向前端开发者解释
* 面向实习面试准备组织内容

生成文档需要允许用户复制、下载和再次编辑。

生成文档不应只存在于内存中，必须保存为 Markdown 文件。

## 八、生成文档页功能逻辑

生成文档页用于查看、切换和导出 AI 生成结果。

页面结构：

* 左侧：文档列表
* 中间：Markdown 源码或结构化目录，可选
* 右侧：Markdown 渲染预览

第一版可只实现文档列表 + Markdown 预览。

文档列表包括：

* 项目概览
* 技术栈分析
* 核心模块解析
* 核心流程说明
* 面试问题与回答
* 简历描述
* 可贡献 PR 方向

预览区必须支持：

* 标题
* 段落
* 引用
* 有序列表
* 无序列表
* 表格
* 行内代码
* 代码块
* 链接

代码块应支持基础高亮。

## 九、历史记录功能逻辑

历史记录用于保存用户分析过的仓库。

历史记录至少包含：

* 仓库名称
* GitHub URL
* 分析时间
* 分析状态
* 生成文档路径
* 核心文件数量
* 是否调用真实 AI
* 错误信息，若存在

用户可以：

* 查看历史分析
* 打开历史生成文档
* 删除历史记录
* 重新分析仓库

第一版可以使用本地 JSON 文件保存历史记录。

第二阶段可使用 SQLite。

## 十一、数据存储规则

### 1. Markdown 文档

生成结果使用 Markdown 文件作为主存储。

推荐目录：

```text
generated_docs/{owner}_{repo}_{timestamp}/
```

文件示例：

```text
01-项目概览.md
02-技术栈分析.md
03-核心模块解析.md
04-核心流程说明.md
05-面试问题与回答.md
06-简历描述.md
07-可贡献PR方向.md
```

Markdown 文件必须可以被普通文本编辑器打开。

### 2. history.json

用于保存历史分析记录。

推荐路径：

```text
data/history.json
```

保存内容：

* id
* repo_url
* owner
* repo
* status
* created_at
* completed_at
* docs_dir
* core_files
* error_message

### 3. config.json

用于保存用户可修改配置。

推荐路径：

```text
data/config.json
```

保存内容：

* backend_base_url
* show_tool_logs
* markdown_font_size
* code_font_size
* max_core_files
* max_file_size
* mock_mode

### 4. temp_repos

用于保存临时 clone 的仓库。

推荐路径：

```text
temp_repos/
```

临时仓库可以被清理。

不要把 clone 下来的仓库提交到 GitHub。

## 十二、模型配置规则

第一版后端只保留一个主要模型：

* 仓库分析模型

仓库分析模型用于：

* 项目概览生成
* 技术栈分析
* 核心模块解析
* 面试问答生成
* 简历描述生成
* PR 方向生成

后续可拆分为：

* 核心文件筛选模型
* 项目分析模型
* 面试问答模型
* 多轮问答模型

第一版不要求支持用户在前端添加多个供应商。

API Key 必须通过后端环境变量读取。

---

## 十三、错误处理规则

所有后端错误都应返回结构化错误。

错误类型包括：

* GitHub URL 无效
* 仓库不存在
* clone 失败
* 仓库过大
* 文件读取失败
* AI API Key 未配置
* AI 调用失败
* 生成文档保存失败

前端不应因为错误崩溃。

前端应显示低打扰错误提示，并保留当前页面状态。

没有 API Key 时，系统应支持 mock 模式，确保演示流程可跑通。

## 十四、UI 视觉规则

本项目 UI 使用：

* Vue 3
* TypeScript
* Vite
* Naive UI

整体视觉风格：

* 浅色主题
* 简洁、干净、低饱和
* 类 AI 工作台 / SaaS Dashboard 风格
* 避免传统后台管理系统过重视觉
* 不使用高饱和大色块
* 不使用过多阴影
* 保持清晰留白

Naive UI 使用要求：

* 使用 `n-layout` 构建整体页面
* 使用 `n-card` 展示工作区块
* 使用 `n-steps` 展示 Agent 执行过程
* 使用 `n-tree` 展示仓库目录
* 使用 `n-tabs` 切换文档
* 使用 `n-tag` 标记文件类型和状态
* 使用 `n-alert` 展示错误和说明
* 使用 `n-spin` 展示分析中状态
* 使用 `n-message` 展示轻量反馈

页面不应只有一个聊天输入框。

必须体现“文件树 + Agent 过程 + 文档预览”的产品结构。

## 十五、开发注意事项

* 所有 AI 生成内容都应允许用户复制和继续编辑
* 生成结果必须存成 Markdown 文件
* 前端不要直接调用 OpenAI API
* API Key 不要提交到 GitHub
* 后端文件读取必须限制大小
* 不要读取二进制文件
* 不要把整个仓库一次性塞给模型
* 模型输出必须尽量结构化
* Prompt 必须要求引用文件路径
* 信息不足时必须输出“不确定”
* Agent 步骤必须可视化
* mock 模式必须保留，保证无 API Key 时可演示
* 第一版优先跑通完整链路，不追求复杂 Agent 框架

## 十六、MVP 功能边界

第一版必须完成：

* Vue + Naive UI 前端骨架
* FastAPI 后端骨架
* GitHub URL 解析
* GitPython clone 仓库
* 目录树生成
* 核心文件筛选
* 基础文件读取
* OpenAI SDK 调用封装
* mock AI 模式
* Markdown 文档生成
* 前端展示 Agent 步骤
* 前端展示文件树
* 前端展示 Markdown 预览

第一版不做：

* 登录注册
* 多用户系统
* 私有仓库授权
* 复杂数据库
* RAG
* MCP
* 自动提交 PR
* 在线 IDE
* 复杂多 Agent
* 支付系统

## 十七、后续扩展方向

MVP 完成后可扩展：

* SSE 流式展示分析过程
* SQLite 保存历史记录
* RAG 支持代码问答
* MCP Server 封装仓库读取工具
* 支持上传本地 zip 项目
* 支持指定分支分析
* 支持 GitHub Issue 筛选和 PR 方向推荐
* 支持导出为 PDF
* 支持多模型供应商

扩展时必须保证核心定位不变：

CodebaseCoach 是开源项目学习与面试准备 Agent，不是普通聊天软件，也不是完整 IDE。
