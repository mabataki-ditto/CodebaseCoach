# 规则筛选 vs RAG — 项目竞争力分析

## 为什么不用 RAG，用规则筛选？

这是个设计选择，不是缺陷。面试官如果问"为什么不用 RAG"，回答如下：

> "RAG 需要先 embedding 全仓库文件，对 451 文件的仓库就是 451 次额外 LLM 调用。我的场景是冷启动分析——用户给一个陌生 GitHub 链接，要在 2 分钟内出结果。如果筛选阶段就要花 1 分钟 embedding + 检索，总耗时翻倍。我用规则筛选的真正原因是：**文件结构本身就是一种天然索引**——入口文件、核心目录、命名约定，这些信息比向量相似度更可靠，而且零成本。RAG 更适合语义搜索场景（'帮我找所有处理认证的代码'），不适合'这个仓库最核心的文件是哪些'这种场景。"

这段话能展示三个能力：**成本意识、场景分析、方案对比**。

## 这个项目会不会像玩具？

**不会**——如果描述对方向。面试官看简历项目，看的是：

| 面试官关心的 | 本项目有没有 |
|---|---|
| 解决了真实问题 | 有：GitHub 仓库代码审查太耗时，AI 辅助但成本不可控 |
| 有工程深度 | 有：10 步工作流 + 工具白名单 + 权限审计 + MCP 集成 |
| 有量化结果 | 有：10 仓库实测，压缩比、耗时、步骤数全部可查 |
| 有技术难点 | 有：MCP stdio 客户端手写、多线程防死锁、协作式取消 |
| 能讲清楚为什么这么做 | 有：规则筛选 vs RAG 的 trade-off 分析 |

**会让面试官觉得像玩具的项目**通常是：调个 API 就完事、没有自己写的核心逻辑、没有量化指标、没有异常处理。本项目全都有。

## 真正有竞争力的是哪几个点

按面试杀伤力排序：

1. **MCP stdio 客户端**（手写协议，多线程防死锁，零 SDK 依赖）—— 面试官会觉得你懂底层协议
2. **工具白名单 + 4 级权限审计**（安全设计）—— 面试官会觉得你有安全意识
3. **确定性评估器**（4 维度，零额外 LLM 成本）—— 面试官会觉得你懂"AI 不能自我评估"这个坑
4. **10 步可观测工作流**（双日志 + 30+ 指标落盘）—— 面试官会觉得你懂工程化
5. **规则筛选 + 成本分析**（不是简单"调个 API"）—— 面试官会觉得你懂 trade-off

## 简历里怎么描述能最大化竞争力

**不要写**："使用规则筛选文件，然后调用 LLM 生成文档"（听起来像玩具）

**应该写**：

> 设计 **10 步骤 Agent 工作流**，通过 **5 级加权打分**（文件名/入口路径/核心目录/扩展名）将任意 GitHub 仓库压缩至 ~10.8% 上下文（实测 10 仓库），**零额外 LLM 成本**；手写 **MCP stdio 客户端**（JSON-RPC 2.0，多线程防死锁）注入协作上下文；实现 **4 维度确定性评估器**（引用准确率/覆盖率/幻觉风险/可用性）对生成结果做质量审计；**工具白名单 + 4 级权限**防止 LLM 越权调用；全程 30+ 指标落盘 JSONL 可回归。

## 总结

**规则筛选不是弱点，是面试时的得分点**——前提是能讲清楚"为什么在这个场景下不用 RAG"。面试官面过太多"调个 API 就说是 AI 项目"的简历，本项目的工程深度反而是稀缺的。

## 上下文压缩比 10.8% 算什么水平？

### 10.8% 的含义

原始候选文本（306KB）→ 经打分筛选后最终进入 LLM 上下文（27KB），只保留了 10.8%，即丢掉了 89.2% 的内容。

### 横向对比

| 策略 | 典型压缩比 | 质量代价 |
|---|---|---|
| 直接把整个仓库塞进 LLM | 100%（不压缩） | 爆 token 上限 / 天价成本 |
| 随机选 12 个文件 | 取决于仓库大小，不可控 | 可能漏掉核心文件 |
| 简单按文件名匹配 | 不可预测 | 经常选错文件 |
| RAG 向量检索 + 分块 | 通常 10-30% | 需要额外 embedding 成本 |
| GitHub Copilot 上下文窗口 | 通常 5-15% | 闭源，不可复现 |
| **本项目（5 级打分筛选）** | **10.8%**（硬约束 12×12KB） | 零额外 LLM 成本，可解释 |

### 为什么 10.8% 是好的

1. **不是"压缩算法"的压缩，是"筛选算法"的压缩**——丢掉的 89.2% 是真正不重要的文件，不是把重要内容截断。451 个候选文件的大仓库（CountBot），压缩到 0.7% 依然能生成 7 份文档，说明筛选是有效的。
2. **硬约束保证了成本可控**——12 文件 × 12KB = 144KB 上限，无论仓库多大，LLM 成本是固定的。
3. **筛选是可解释的**——5 级打分（文件名优先级 → 入口路径 → 入口命名 → 核心目录 → 扩展名白名单），每个文件都有打分理由，不是黑盒。
4. **与 RAG 方案对比**：RAG 需要先 embedding 所有文件（额外 LLM 成本），本方案用纯规则打分，零额外成本达到接近的效果。

### 一句话总结

**10.8% 在"零额外 LLM 成本的规则筛选"方案中是优秀水平**。

## 零额外 LLM 成本 vs 有额外 LLM 成本

### 零额外 LLM 成本 = 用纯规则筛选，不调 LLM

`file_selector_service.py` 的核心逻辑是纯 Python 规则：

1. 遍历仓库文件列表，用 `IGNORED_NAMES`（13 类目录）过滤
2. 对剩余文件按 5 级加权打分
3. 取 Top-12，每文件截断到 12KB

整个过程：**0 次 LLM 调用，0 token 消耗**。筛选速度取决于文件系统 I/O，不花钱。

### 有额外 LLM 成本 = 用 LLM 做筛选

典型 RAG 方案需要额外调用 LLM：

```
仓库文件 → embedding 模型（LLM 调用 #1）→ 向量库 → 用户 query → embedding（LLM 调用 #2）
→ 向量检索 → 重排序模型（LLM 调用 #3）→ 最终上下文
```

对 451 个文件的仓库，embedding 就要 451 次调用。即使用小模型（如 text-embedding-3-small），每次分析多花几分钱。

更激进的方案是让 LLM 自己决定要看哪些文件（ReAct 模式），那每次"看文件列表 → 决定选哪个 → 读文件内容"都算一次 LLM 调用，步骤数会爆炸。

### 用项目实际数据说明

从 `metrics.jsonl` 可以看到，本项目每次分析 LLM 调用固定为 7 次（对应 7 份文档生成），没有因为筛选环节多花 1 次调用。如果换成 RAG 方案，同样的 451 文件仓库至少多花 451 次 embedding 调用。

### 一句话总结

- **零额外 LLM 成本**：筛选阶段不调 LLM，用纯规则打分，token 消耗为 0
- **有额外 LLM 成本**：筛选阶段调 embedding/reranker/LLM 来决定选哪些文件，相当于为"挑选"这件事本身花钱
